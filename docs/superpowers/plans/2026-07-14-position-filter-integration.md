# 직무 필터 API 연동 구현 계획

> **목표:** 리더 DB의 `job_category` 데이터를 정본으로 사용해 직무를 선택했을 때 해당 공고가 정상 조회되도록 한다.

## 안전 원칙

- DB 스키마·데이터·백엔드 조회 로직은 변경하지 않는다.
- 카드에 표시되는 기존 직무 추론 로직과 다른 검색 필터는 건드리지 않는다.
- 직무 선택지는 `/api/v1/job-categories` 응답의 실제 `name`만 사용한다.
- 카테고리 API가 비었거나 실패하면 임의 직무값을 만들지 않고 `전체`만 제공한다.
- `AGENTS.md`와 작업 범위 밖 파일은 커밋하지 않는다.

## 작업 순서

1. frontend/backend의 현재 변경을 파일별로 확인하고 의도한 작업만 보존한다.
2. 두 저장소에 최신 `origin/main`을 통합하고 충돌 시 최신 API 계약을 우선한다.
3. 직무 카테고리 요청에 현재 채용 풀(`domestic`/`global`)이 전달되고, 응답의 정확한 `name`이 필터 파라미터로 쓰이는 테스트를 먼저 추가한다.
4. `placeholders.tsx`와 필요한 API 래퍼만 최소 수정한다.
5. 프론트 단위 테스트·빌드와 실제 로컬 API 요청을 검증한다.
6. 변경 파일을 명시적으로 스테이징해 커밋하고 기능 브랜치를 푸시한다.

## 완료 조건

- 최초 조회에는 전체 공고가 표시된다.
- 직무 선택 시 URL/API의 `position` 값이 DB 카테고리명과 동일하다.
- 국내/글로벌 전환 시 해당 풀의 카테고리 목록을 다시 불러온다.
- 직무를 `전체`로 되돌리면 `position` 조건이 제거된다.
- 기존 기술·지역·마감일 필터 테스트와 프로덕션 빌드가 통과한다.
